# 人事管理系统（HRMS）需求文档（完整版）
## 一、文档概述
### 1.1 文档目的
本文档全面定义人事管理系统（HRMS）的业务场景、功能边界、数据规范、非功能要求及验收标准，为数据库设计、系统开发、测试验收提供统一依据，确保系统符合企业级人事管理的流程化、规范化、可扩展需求，同时满足数据安全与合规要求。

### 1.2 文档范围
覆盖系统核心模块（组织结构与人员管理、考勤与请假管理、绩效考核管理）、扩展模块规划、权限体系、数据模型、非功能需求等，不包含具体技术实现细节（如框架选型、接口开发规范），聚焦“what to do”而非“how to do”。

### 1.3 目标用户与使用场景
| 用户角色 | 核心使用场景 |
|----------|--------------|
| 系统管理员 | 系统初始化配置、角色权限分配、数据备份恢复、用户账号管理 |
| 人力资源人员 | 员工档案维护、组织架构调整、考勤规则配置、绩效指标设计、流程审批、数据统计 |
| 部门负责人 | 部门人员管理、下属请假/绩效审批、部门考勤与绩效数据查看 |
| 普通员工 | 个人信息查询、请假/补卡/绩效自评提交、考勤记录查看、绩效结果查看 |
| 扩展角色（薪酬专员/招聘专员） | 薪酬核算、招聘流程管理等扩展场景操作 |

## 二、项目背景与总体目标
### 2.1 项目背景
随着企业规模扩大与数字化转型深入，人事管理已从传统的人工登记、分散管理，升级为**以数据为核心的流程化、规范化、智能化管理**。人事管理系统（HRMS）作为企业核心的**主数据系统（Master Data System）**，其数据库设计的合理性、稳定性与扩展性，直接决定了系统对复杂业务场景的支撑能力、数据可信度，以及后续与财务、ERP等系统的集成效率。当前企业面临员工规模增长、业务流程精细化、合规要求提升等挑战，亟需一套具备企业级架构思维的数据库模型，实现人事数据的全生命周期管理与业务规则的刚性约束。

### 2.2 项目目标
1. 实现复杂业务规则的数据库级建模，确保核心逻辑（时间重叠校验、状态流转、数据唯一性）不依赖应用层，提升数据正确性；
2. 针对高频场景（考勤查询、绩效统计）优化数据结构，平衡一致性与性能；
3. 预留清晰扩展边界，支持薪酬核算、培训管理等模块无缝接入；
4. 满足数据合规要求，实现核心数据可追溯、不可篡改，支持审计核查；
5. 构建精细化权限体系，实现功能与数据范围的精准管控。

## 三、设计原则与总体约束
### 3.1 核心设计原则
- **数据库主导业务正确性**：关键业务规则（如时间重叠、状态流转）通过数据库约束（主键、外键、唯一索引、触发器、CHECK约束）实现；
- **强一致性优先**：核心数据操作满足ACID事务特性，确保并发场景下数据完整性；
- **历史可追溯，不可篡改**：已生效业务数据仅支持逻辑删除，关键变更记录日志，历史数据永久保留；
- **面向扩展设计**：模块化、松耦合数据结构，核心表预留扩展字段，关联关系避免硬编码；
- **数据轻量化与冗余可控**：高频访问数据适度冗余，非核心数据通过“主表+扩展表”分离存储。

### 3.2 数据生命周期约束
- 核心数据（员工档案、组织架构、考勤/绩效记录）仅支持逻辑删除（`is_deleted`字段标识），不允许物理删除；
- 重要业务对象（员工、请假申请、绩效周期）需具备明确生命周期状态，状态值来自预定义枚举；
- 关键状态变更（员工离职、请假批准）需记录日志（变更人、时间、前后值），日志不可修改；
- 历史数据归档：超过3年的低频数据归档至历史库，支持按需回溯，提升主库性能。

## 四、系统角色与权限模型需求（RBAC）
### 4.1 角色体系
系统采用“基础角色+自定义角色”体系，基础角色覆盖核心场景，自定义角色支持按需扩展：

| 角色类型 | 核心角色 | 角色说明 |
|----------|----------|----------|
| 系统级角色 | 系统管理员 | 全局配置、角色权限管理、数据备份恢复、用户账号生命周期管理 |
| 人事专业角色 | 人力资源人员 | 人事主数据维护、规则配置、流程审批、报表统计 |
|  | 薪酬专员（扩展） | 薪酬核算、薪资发放记录维护，仅访问薪酬相关数据 |
|  | 招聘专员（扩展） | 招聘流程管理、候选人数据维护，仅访问招聘相关模块 |
| 管理角色 | 部门负责人 | 部门及下级人员管理、审批（请假/调动/绩效）、数据查看 |
|  | 跨部门负责人（扩展） | 跨部门项目团队审批与绩效评估，访问指定团队数据 |
| 普通角色 | 普通员工 | 个人数据查询、自助申请提交（请假/补卡等） |
|  | 绩效评估人（临时） | 特定绩效周期内对指定员工评分，周期结束权限失效 |

### 4.2 权限设计要求
- 权限模型：RBAC v3，支持“用户-角色-权限”三级映射+“用户-权限”直接映射；
- 权限粒度：“操作-资源”级，核心权限包括功能操作（查询/新增/修改/审批等）、数据资源（员工/考勤/绩效数据等）；
- 权限规则：多角色权限叠加，支持权限禁用/启用，权限变更记录日志；
- 数据范围控制：“组织树+指定范围”双模式，指定范围优先级高于组织树，支持敏感数据脱敏（如身份证号前6后4位显示）。

### 4.3 权限校验要求
- 实时性：权限变更立即生效，不依赖缓存；
- 一致性：功能权限与数据范围权限联动校验，无越权操作可能；
- 审计性：所有权限访问操作记录日志，包含操作人、时间、资源、结果。

## 五、核心功能需求
### 5.1 模块一：组织结构与人员管理
#### 5.1.1 组织结构管理
##### 业务需求
- 支持无限层级树形结构，节点对应公司/部门/团队/项目组；
- 核心属性：组织编码（唯一）、名称、类型、上级组织、负责人、生效/失效时间、状态（启用/禁用）；
- 支持动态调整：新增、修改、迁移、合并、禁用（逻辑删除）；
- 支持虚拟分组：同一组织可归属多个虚拟分组（如按业务线/区域）。

##### 业务规则
- 禁止循环引用，数据库层约束防止；
- 组织编码全局唯一，规则可配置（如“公司编码-部门编码”）；
- 禁用节点时同步禁用下属有效节点；
- 合并节点时员工自动迁移，历史业务数据关联原节点；
- 生效时间≤失效时间，未生效节点不可分配员工，过期节点自动禁用。

##### 验收标准
- 单条SQL可查询任意节点的完整上下级链路；
- 结构调整不破坏历史数据，员工归属自动同步；
- 并发调整无数据冲突，禁用节点不可分配员工。

##### 数据属性
| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| org_id | VARCHAR(32) | 主键 | 组织唯一标识（UUID） |
| org_code | VARCHAR(50) | 唯一、非空 | 组织编码，按规则生成 |
| org_name | VARCHAR(100) | 非空 | 组织名称 |
| org_type | VARCHAR(20) | 非空 | 枚举：公司/部门/团队/项目组 |
| parent_org_id | VARCHAR(32) | 可空 | 上级组织ID（顶级节点为空） |
| manager_emp_id | VARCHAR(32) | 可空 | 负责人工号（关联员工表） |
| status | VARCHAR(20) | 非空 | 枚举：启用/禁用 |
| effective_time | DATETIME | 非空 | 生效时间 |
| expire_time | DATETIME | 可空 | 失效时间（永久有效为空） |
| is_deleted | TINYINT(1) | 非空，默认0 | 逻辑删除标识（0=未删，1=已删） |
| create_time/update_time | DATETIME | 非空 | 记录创建/更新时间 |
| create_by/update_by | VARCHAR(32) | 非空 | 记录创建/更新人（用户ID） |

#### 5.1.2 员工档案管理
##### 业务需求
- 员工信息分核心信息（必填）与扩展信息（可选）：
  - 核心信息：工号、姓名、身份证号、性别、出生日期、联系方式、入职日期、组织归属、岗位、雇佣类型、员工状态；
  - 扩展信息：教育背景、工作经验、证书、紧急联系人、银行账户、薪酬等级、社保公积金信息；
- 支持批量导入/单个修改/批量更新，导入数据格式校验；
- 支持附件（照片、简历）上传存储，唯一关联员工；
- 支持信息变更申请流程，审批后生效。

##### 业务规则
- 唯一标识：工号、身份证号全局唯一（数据库唯一索引约束）；
- 员工状态枚举：试用期/在职/待离职/离职/停薪留职，状态流转符合逻辑顺序；
- 组织归属必须是启用状态的组织节点；
- 雇佣类型与状态关联：实习员工不可为“正式在职”，离职员工雇佣类型锁定为“离职”；
- 核心信息修改记录日志，扩展信息可选择性记录。

##### 历史追溯要求
- 关键信息变更（组织、岗位、状态、雇佣类型）记录历史表（`employee_history`）；
- 历史表支持按时间范围查询任意时段员工信息状态；
- 历史记录不可修改，员工离职后永久保留；
- 支持信息版本回溯，查看任意时间点完整档案。

| phone | VARCHAR(20) | 非空 | 手机号（脱敏存储） |
| email | VARCHAR(100) | 唯一 | 邮箱 |
| hire_date | DATE | 非空 | 入职日期 |
| org_id | VARCHAR(32) | 非空 | 所属组织ID（关联组织表） |
| position | VARCHAR(50) | 非空 | 岗位名称 |
| employment_type | VARCHAR(20) | 非空 | 枚举：正式/实习/劳务派遣 |
| emp_status | VARCHAR(20) | 非空 | 枚举：试用期/在职/待离职/离职/停薪留职 |
| manager_emp_id | VARCHAR(32) | 可空 | 直接上级工号（关联员工表） |

| create_time/update_time | DATETIME | 非空 | 记录创建/更新时间 |
| create_by/update_by | VARCHAR(32) | 非空 | 记录创建/更新人 |

### 5.2 模块二：考勤与请假管理
#### 5.2.1 请假管理（时间区间约束重点）
##### 业务需求
- 请假类型：事假、病假、年假、婚假等，支持类型配置（新增/禁用/改名）；
- 申请信息：类型、开始时间（精确到时分）、结束时间、事由、附件（病假证明等）；
- 支持分段请假（多不连续时间段），需分别校验重叠；
- 多级审批流程（员工→直接上级→部门负责人→HR），节点可配置；
- 请假额度管理（如年假10天/年），申请时自动校验额度充足性。

##### 核心业务规则
- 时间重叠校验：同一员工有效请假记录（已批准/审批中）时间区间不得重叠（部分/完全重叠）；
- 重叠校验实现：`emp_id`+`leave_start_time`+`leave_end_time`复合索引+触发器，绕过应用层也拒绝非法数据；
- 时间逻辑：结束时间>开始时间，单次请假时长不超过配置上限（如事假15天）；
- 额度校验：有额度限制的请假类型（年假/调休假）申请时长≤剩余额度；
- 状态枚举：待审批/审批中/已批准/已拒绝/已撤销/已作废，流转符合规则。

##### 状态流转规则
| 当前状态 | 允许下一个状态 | 触发操作 | 约束条件 |
|----------|----------------|----------|----------|
| 待审批 | 审批中 | 提交审批 | 信息完整，额度充足 |
| 待审批 | 已撤销 | 员工撤销 | 仅申请人操作，审批未开始 |
| 审批中 | 已批准 | 所有节点通过 | 无重叠请假，额度充足 |
| 审批中 | 已拒绝 | 任意节点拒绝 | 需填拒绝理由 |
| 审批中 | 已撤销 | 员工撤销 | 审批未完成 |
| 已批准 | 已作废 | HR作废 | 需填作废理由，仅HR操作 |
| 已拒绝 | 待审批 | 重新提交 | 修正信息后可再次提交 |

##### 历史追溯与数据约束
- 已批准记录不可修改/删除，调整需“作废原记录+新增新记录”；
- 审批日志记录每个节点的审批人、时间、意见、结果；
- 额度自动扣减/恢复：批准时扣减，拒绝/撤销时恢复；
- 数据分表：请假主表（`leave_apply`）+时间段表（`leave_time_segment`），支持分段请假。

##### 数据属性（主表+时间段表）
| 表名 | 字段名 | 数据类型 | 约束 | 说明 |
|------|--------|----------|------|------|
| leave_apply | leave_id | VARCHAR(32) | 主键 | 请假申请唯一标识 |
|  | emp_id | VARCHAR(32) | 非空，外键 | 申请人工号（关联员工表） |
|  | leave_type | VARCHAR(20) | 非空 | 请假类型（关联配置表） |
|  | apply_status | VARCHAR(20) | 非空 | 申请状态 |
|  | apply_time | DATETIME | 非空 | 提交时间 |
|  | reason | TEXT | 可空 | 请假事由 |
|  | attachment_url | VARCHAR(255) | 可空 | 附件路径 |
|  | total_days | DECIMAL(10,2) | 非空 | 总天数（含小数） |
|  | create_by/update_by | VARCHAR(32) | 非空 | 操作人 |
|  | create_time/update_time | DATETIME | 非空 | 操作时间 |
| leave_time_segment | segment_id | VARCHAR(32) | 主键 | 时间段唯一标识 |
|  | leave_id | VARCHAR(32) | 非空，外键 | 关联请假主表ID |
|  | leave_start_time | DATETIME | 非空 | 时间段开始时间 |
|  | leave_end_time | DATETIME | 非空 | 时间段结束时间 |
|  | segment_days | DECIMAL(10,2) | 非空 | 该段天数 |

##### 验收标准
- 并发提交重叠请假仅一条通过，其余拒绝；
- 手动修改数据库导致重叠时，触发器拦截报错；
- 额度实时更新，状态流转符合规则，无非法修改可能。

#### 5.2.2 日常考勤管理（高频数据场景）
##### 业务需求
- 考勤类型：打卡（上班/下班）、外勤、加班、补卡；
- 多终端打卡：手机APP、考勤机、web端，数据自动同步；
- 状态自动推导：根据打卡时间、请假/加班记录，生成正常/迟到/早退/旷工等状态；
- 人工补卡：支持补卡申请，审批后生效，每月补卡次数有上限；
- 异常处理：员工可对异常记录申诉，HR审核后调整状态。

##### 数据规则
- 打卡约束：同一员工每日同一类型（上班/下班）最多2条记录（正常+补卡），补卡需在截止时间后；
- 时间逻辑：上班打卡时间≤下班打卡时间，外勤需填地点与事由；
- 状态推导规则可配置（如9:00后上班为迟到）；
- 补卡规则：打卡日期后3个工作日内提交，逾期不可补卡。

##### 性能要求
- 高频查询响应：单条件查询≤1秒，多条件组合查询≤2秒；
- 大数据量支持：10000+员工每日考勤存储，年数据360万+时统计查询性能稳定；
- 索引优化：`emp_id`+`attendance_date`+`attendance_status`复合索引；
- 分区策略：按`attendance_date`按月分区，提升查询效率。

##### 数据属性
| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| attendance_id | VARCHAR(32) | 主键 | 考勤记录唯一标识 |
| emp_id | VARCHAR(32) | 非空，外键 | 员工工号（关联员工表） |
| attendance_date | DATE | 非空 | 考勤日期 |
| attendance_type | VARCHAR(20) | 非空 | 枚举：打卡/外勤/加班/补卡 |
| check_in_time | DATETIME | 可空 | 上班打卡时间 |
| check_out_time | DATETIME | 可空 | 下班打卡时间 |
| attendance_status | VARCHAR(20) | 非空 | 枚举：正常/迟到/早退/旷工/请假/外勤/加班 |
| exception_reason | TEXT | 可空 | 异常原因 |
| appeal_status | VARCHAR(20) | 非空，默认“无申诉” | 枚举：无申诉/申诉中/申诉通过/申诉拒绝 |
| is_deleted | TINYINT(1) | 非空，默认0 | 逻辑删除标识 |
| create_time/update_time | DATETIME | 非空 | 操作时间 |

### 5.3 模块三：绩效考核管理
#### 5.3.1 绩效周期与规则配置
##### 业务需求
- 周期类型：月度/季度/半年度/年度，类型可配置；
- 周期属性：名称、类型、开始/结束时间、考核指标集、评分规则、状态（未开始/进行中/已结束/已归档）；
- 指标集支持多维度（如工作业绩60%+工作态度20%+团队协作20%），每个维度含多个具体指标；
- 评分方式：百分制/等级制（A/B/C/D/E）/分数段制（1-5分），与指标绑定；
- 支持不同部门/岗位使用不同指标集。

##### 业务规则
- 周期时间约束：同一组织内同类型周期时间不重叠；
- 指标权重约束：同一指标集维度权重之和=100%，单个指标权重非负（数据库CHECK约束）；
- 状态流转：未开始→进行中→已结束→已归档，已结束周期不可修改指标集与规则；
- 考核对象分配：同一员工同一周期仅属一个指标集。

#### 5.3.2 绩效评估流程
##### 业务需求
- 多阶段评估流程：员工自评→上级评估→跨级评估（可选）→同事评估（可选）→HR审核；
- 阶段约束：每个阶段有明确开始/截止时间，超时自动提醒；
- 评估结果：各指标得分、维度总分、最终总分、文字评价意见；
- 申诉机制：员工对最终结果有异议可提交申诉，HR组织复核。

##### 数据一致性要求
- 阶段提交为原子事务：多指标得分、评价意见需同时写入，失败则回滚；
- 总分计算：按阶段权重汇总（如自评20%+上级评估80%），由数据库视图/存储过程实现；
- 数据锁定：阶段结束后该阶段评分不可修改；
- 最终结果不可篡改：周期结束后仅可通过申诉复核生成新结果，原结果保留。

##### 历史追溯与验收标准
- 评估日志：记录各阶段评分数据、提交人、提交时间；
- 申诉记录：记录申诉理由、时间、复核结果、复核人；
- 验收标准：
  1. 事务一致性：提交失败无中间状态；
  2. 总分计算准确，符合权重规则；
  3. 已结束周期不可直接修改评分；
  4. 历史结果查询响应时间≤2秒。

## 六、扩展模块规划（预留扩展边界）
### 6.1 薪酬核算管理
- 核心需求：基于考勤、绩效结果自动核算薪酬，支持不同岗位薪酬结构（基本工资、绩效工资、奖金、补贴、扣款）；
- 数据关联：员工表（薪酬等级、银行账户）、考勤表（请假扣款）、绩效表（绩效工资系数）；
- 预留设计：员工表预留`salary_grade`（薪酬等级）、`bank_account`（银行账户）字段，考勤表预留`deduction_amount`（扣款金额）字段。

### 6.2 培训管理
- 核心需求：培训计划制定、报名、考勤、评估；
- 数据关联：员工表（参训员工）、组织表（培训组织部门）；
- 预留表结构：`training_plan`（培训计划表）、`training_enrollment`（报名表）、`training_attendance`（培训考勤表），通过`emp_id`/`org_id`关联核心表。

### 6.3 招聘管理
- 核心需求：职位发布、候选人管理、面试安排、入职转化；
- 数据关联：组织表（招聘部门）、员工表（面试官）；
- 预留表结构：`recruitment_position`（职位表）、`candidate`（候选人表）、`interview_arrangement`（面试安排表），通过`org_id`/`emp_id`关联核心表。

## 七、数据安全与合规需求
### 7.1 数据安全要求
- 敏感数据加密：身份证号、手机号、银行账户采用AES加密存储，查询按需解密；
- 访问日志审计：核心数据访问操作（查询/修改/删除）记录日志，包含访问人、时间、内容、IP，日志保留1年；
- 数据库权限控制：应用角色仅分配最小必要权限，避免超权操作；
- 备份与恢复：每日全量备份+每小时增量备份，备份数据异地存储，支持按时间点恢复。

### 7.2 合规要求
- 符合《个人信息保护法》：明确告知员工信息收集用途，员工有权查询/更正个人信息；
- 数据留存：员工离职后档案、绩效记录至少留存2年；
- 审计支持：提供合规审计报表（数据访问日志、权限变更日志），支持导出。

## 八、非功能需求
### 8.1 性能需求
- 响应时间：普通查询≤1秒，复杂统计≤3秒，批量导入≤5秒/1000条；
- 并发支持：支持50+用户同时在线操作，10+用户并发提交申请（请假/补卡）；
- 数据容量：支持10000+员工全量数据存储，3年以上历史数据归档后不影响主库性能。

### 8.2 易用性需求
- 界面设计：布局清晰，导航明确，核心功能入口≤3步操作可达；
- 错误提示：操作错误时给出明确易懂提示（如“密码错误”“请假时间重叠”）；
- 兼容性：支持Chrome、Edge、Firefox主流浏览器，适配1920×1080、1366×768分辨率。

### 8.3 可扩展性需求
- 架构支持：模块化设计，新增模块无需修改核心表结构；
- 规则配置：考勤规则、绩效指标、审批流程支持可视化配置，无需代码开发；
- 集成支持：预留标准化接口，支持与财务、ERP系统数据同步。

### 8.4 可靠性需求
- 数据可靠性：核心数据写入采用事务，避免数据丢失或不一致；
- 系统可用性：全年可用性≥99.9%，故障恢复时间≤1小时；
- 容错性：支持操作回滚，批量导入失败时可恢复至导入前状态。
